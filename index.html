How it works

1) Discord Components v2 ‚Äî rich interactive prompts

Unlock native Discord interactions: buttons, selects, modals, and attachment-backed file blocks.

# openclaw.config.yaml
channels:
  discord:
    enabled: true
    token: ${DISCORD_BOT_TOKEN}
    components:
      version: 2                 # Enable Components v2
      fileBlocks: true           # Attachment-backed file blocks
      modals: true               # Modal prompts
      selects: true              # Select inputs
      buttons: true              # Buttons

prompts:
  - id: "deploy_flow"
    channel: discord
    ui:
      layout: cv2                # CV2 container layout
      blocks:
        - type: select
          id: env
          label: Environment
          options: ["staging", "prod"]
        - type: button
          id: approve
          label: Approve
        - type: fileBlock
          id: artifact
          label: Upload artifact

Why it matters: Bring interactive agent UX directly into Discord with zero context switching.

2) Discord CV2 UI passthrough + exec approval UX

Refined containers, button layout, embeds passthrough, and Discord-forwarding skip for clean approvals.

# openclaw.config.yaml
channels:
  discord:
    ui:
      componentsV2:
        containerStyle: "card"
        buttonLayout: "row"
        embedsPassthrough: true    # Preserve original embeds
        forwardToDiscord: false    # Skip redundant forward when native UI is present
approval:
  exec:
    required: true
    prompts:
      confirm:
        channel: discord
        ui:
          layout: cv2
          buttons:
            - id: yes
              label: "Approve"
              style: success
            - id: no
              label: "Reject"
              style: danger

Why it matters: Clearer, safer run approvals without clutter or double-posting.

3) Plugins: llm_input and llm_output hooks

Observe prompt/input context and model output usage details in your extensions.

// plugin.js
export function on_llm_input(payload) {
  // payload: { prompt, system, tools, user, metadata }
  logger.info({ size: payload.prompt?.length }, "llm_input observed");
}

export function on_llm_output(payload) {
  // payload: { tokens: { prompt, completion, total }, model, latencyMs, text }
  metrics.counter("llm.tokens", payload.tokens.total);
  audit.write({
    model: payload.model,
    latency: payload.latencyMs,
    sample: payload.text?.slice(0, 200)
  });
}

// plugin.manifest.json
{
  "name": "telemetry-plugin",
  "hooks": ["llm_input", "llm_output"]
}

Why it matters: Gain transparent observability for quality, cost, and safety controls.

4) Subagents: nested sub-agents (depth-aware)

Let agents spawn children (and grandchildren) with safe depth and child limits.

# openclaw.config.yaml
agents:
  defaults:
    subagents:
      maxSpawnDepth: 2            # allow sub-sub-agents
      maxChildrenPerAgent: 5      # default cap
      toolPolicy:
        inherit: true
        depthAware: true
  registry:
    orchestrator:
      entry: orchestrator.js

# orchestrator.js
export default async function orchestrator(ctx) {
  const planner = await ctx.spawn("planner");     // depth 1
  const exec = await planner.spawn("executor");   // depth 2
  return await exec.run({ task: ctx.input });
}

Why it matters: Compose complex workflows while bounding risk and cost.

6) Cron/Gateway: finished-run webhook + auth token

Toggle finished-run notifications and secure outbound posts with a dedicated token.

# openclaw.config.yaml
cron:
  schedules:
    - id: nightly-report
      cron: "0 2 * * *"
      workflow: nightly
      notify: true                 # deliver finished-run webhook
  webhookToken: ${CRON_WEBHOOK_TOKEN}  # dedicated auth for outbound posts

gateway:
  webhooks:
    finishedRun:
      url: https://example.com/hooks/openclaw/finished
      auth: header
      headerName: "X-OpenClaw-Token"
      headerValue: ${CRON_WEBHOOK_TOKEN}

Why it matters: Simple, secured integrations with your pipelines and dashboards.

7) Channels: deduplicated probe/token base types

Unified probe/token resolution across core and extensions while preserving per-channel error typing.

// channels.config.json
{
  "probe": { "timeoutMs": 3000, "retry": 1 },
  "tokenResolution": { "source": "env", "key": "CHANNEL_TOKEN" }
}

// Extension can rely on shared base types
import { BaseProbe, TokenResolver } from "@openclaw/channels-base";

export class MyChannelProbe extends BaseProbe {
  async check() { return this.ping("https://api.mychannel.dev/health"); }
}

export const resolver = new TokenResolver({ envKey: "MYCHANNEL_TOKEN" });

Why it matters: Less duplication, clearer errors, and faster extension development.
